# 后续需求与功能开发计划

本文档以 2026-07-11 的仓库实现为基线，用于将现有 backlog 转换为可开发、可验收的版本计划。当前项目已具备多语言博客、账户与权限、文章/分类/标签、MinIO 特色图上传、收藏/点赞/关注/通知、评论提交与后台审核等能力。

> **排期假设**：1 名全栈开发为主、每迭代 2 周，设计/测试按需支持。日期是建议窗口；每个迭代开始前根据线上数据、风险和人力重新确认范围。

## 一、产品目标与范围

未来 4 个月优先把项目从“功能齐备的博客”推进为“可持续运营的内容社区”：

1. 让认证、权限和发布链路能安全稳定地运行在生产环境。
2. 让读者评论、作者回复、审核和通知形成完整闭环。
3. 让图片资产与草稿编辑可追踪、可恢复，降低内容丢失和存储失控风险。
4. 提升内容发现、搜索和基础运营能力，并建立可观测、可恢复的运维基线。

本周期不做协作编辑、多租户、插件化和独立搜索集群。这些能力的架构与成本都明显高于当前用户规模的收益，待内容量、并发和运营需求有数据支撑后再立项。

## 二、优先级原则

- **P0：安全与可靠性**。生产密钥、会话、服务端 RBAC、数据迁移、备份和回归测试先于新增界面。
- **P1：内容闭环**。评论、审核、通知、媒体和草稿恢复直接影响读者与作者的日常使用。
- **P2：增长与运营**。搜索、RSS、站点地图、仪表盘以真实内容量和运营目标为前提。
- **P3：效率增强**。导入导出、批量操作、版本历史等在核心流程稳定后按使用频率投入。

## 四、重点需求说明

### R0：质量基线

- JWT 默认密钥拦截和标签批量读取已补回归测试，避免后续重构悄然回退。
- `CORS_ORIGIN`、`RATE_LIMIT_WINDOW`、`RATE_LIMIT_MAX_REQUESTS` 已接入业务代码并覆盖测试；未读取的缓存/本地上传模板变量已移除。
- 当前以 Node 原生测试运行器 + `tsx` 加载器覆盖 R0；CI 已执行格式、类型、API 路由包装、R0 回归和 lint。Playwright 端到端覆盖留作 R1 前的独立任务，避免在未确定浏览器测试基础设施前引入依赖。

### R1：会话与 RBAC

- **已落地**：Cookie 契约（`blog_refresh_token` / Path=`/api/auth` / SameSite=Lax）、登录写入与刷新轮换、登出清理、Cookie 路径 Origin 防护；前台 `AuthProvider` 不再保存 refresh。
- **已落地**：分类/标签写接口 `requireTaxonomyManager`（author/admin/super_admin）与管理页 `TaxonomyManageGate`。
- **已落地**：`POST /api/auth/refresh` 仅 Cookie；登录/刷新 JSON 不再返回 `refreshToken`（Body 兼容已移除）。
- **已落地**：统一客户端请求层——`lib/utils/client-bearer-auth.ts`（token 读写唯一入口）、`lib/utils/client-api-fetch.ts`（`clientApiFetch` / Cookie 刷新单飞）、全局 401 拦截先刷新重试再登出；Bearer 短期兼容保留。

### R2：评论闭环

- **已落地（第一切片）**：`GET /api/comments?postId=` 仅返回 `approved` 树（最多两层）；`POST` 支持 `parentId`；前台 `PostComments` 列表/回复；公开载荷不含邮箱/IP。
- **已落地（第二切片）**：回复/新评论写入 `user_notifications`（通知被回复者与文章作者）；审核 `approved`/`spam` 通知评论作者；文案三语；与个人中心通知 CTA 的 `postSlug`/`commentId` 约定对齐。
- **已落地（第三切片）**：删除父评论软删 `deleted` 占位 + 前台「原评论已删除」；审核删除写审计；管理页 `clientApiFetch`；`comment-spam` 规则增强；迁移 `0009_serious_sumo.sql`。
- 数据模型复用 `comments.parentId`；后续确有长讨论需求再扩展深层树和分页策略。
- **可选后续**：邮件频控与邮件通知；更复杂垃圾分类。

### R3：媒体与编辑可靠性

- `media` 成为上传元数据的唯一来源；新增媒体引用模型记录文章特色图、正文引用等关系。对象键、MIME、尺寸、大小、上传者和可访问 URL 均可查询。
- 删除采用“解除引用 -> 延迟清理 -> 定时对账”三步，不能在替换图片时直接删除可能被复用的对象。
- 自动保存以草稿版本号为条件更新；页面显示最近保存时间、失败状态和冲突处理入口。文章版本历史可作为本版本的预研或下一版本需求，不与自动保存捆绑承诺。

### R4：发现与运营

- 当前规模优先 MySQL FULLTEXT，不引入 Elasticsearch。搜索支持关键词、分类、标签、发布时间和排序，并保留现有 LIKE 查询作为降级策略。
- SEO 交付包含 RSS、sitemap、robots、canonical 与分享元数据；发布和下线文章需触发相应更新。
- 仪表盘首期只做已具备数据的浏览、点赞、收藏、分享、评论待审和订阅趋势；每个指标必须说明统计范围、时区和更新时间。

## 五、横向工程要求

每一项功能需求都应同时完成以下内容，不能只完成页面：

1. 明确用户角色、成功/失败状态和验收场景。
2. 评估数据库迁移、索引和旧数据回填；迁移必须可回滚或有恢复方案。
3. 定义 API 请求/响应类型、鉴权和限流规则，并同步内置 API 文档。
4. 补齐中英日三语文案、无障碍标签和移动端交互。
5. 增加对应单元、集成或端到端测试；更新功能文档、TODO 与变更记录。
6. 对上传、权限、审核、批量删除等操作写入结构化日志和必要审计记录。

## 六、迭代与验收机制

- 每个迭代第 1 天完成需求澄清、接口/表结构设计和验收样例；中期进行一次联调演示；结束前保留至少 2 天回归和文档更新。
- 需求进入开发前必须具备：目标用户、范围边界、原型或交互说明、数据影响、权限规则和验收标准。
- 需求完成的统一定义：代码审查通过，相关检查通过，API/数据库/部署文档同步，关键流程在预发布环境验证，并能从日志定位失败原因。
- 每月复盘一次：评论通过率与审核时效、草稿恢复成功率、搜索使用率、上传失败率、认证失败率和接口错误率。数据不足时，下一月优先补埋点与可观测性，而非扩展功能面。

## 七、风险与决策点

| 风险                                         | 处理策略                                       | 需要确认的决策                                                   |
| -------------------------------------------- | ---------------------------------------------- | ---------------------------------------------------------------- |
| Cookie 会话迁移影响大量直接读取 token 的组件 | 先抽统一客户端请求层，分阶段切换并保留短期兼容 | 是否存在独立前端域名或跨站嵌入场景，决定 `SameSite` 与 CORS 策略 |
| 媒体对象被替换或复用后误删                   | 媒体引用表、引用计数、延迟清理和定期对账       | 是否允许文章正文复用同一媒体、是否支持私有媒体                   |
| 评论增多带来垃圾内容与通知噪声               | 审核队列、限流、关键词规则、频控和举报预留     | 首期是否保持匿名评论；审核 SLA 由谁承担                          |
| 搜索规模超过 MySQL 能力                      | 先通过索引、查询日志和响应时间验证             | 达到内容量/查询量阈值后再评估独立搜索服务                        |

## 八、后续候选项（不排入本周期）

- 个性化推荐、阅读进度、分享渠道归因和订阅分群。
- 协作编辑、工作区/多租户、主题或插件系统。
