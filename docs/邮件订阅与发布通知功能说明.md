# 邮件订阅与发布通知功能说明

## 1. 背景与目标

本功能用于实现博客文章发布后的邮件通知能力，满足以下业务要求：

- 未登录用户可输入邮箱进行订阅；**须先收取邮箱验证码，校验通过后才写入订阅**；取消订阅同样须邮箱验证码。
- 已登录用户无需输入邮箱，直接使用注册邮箱进行订阅/取消订阅（**请求须携带 JWT，服务端校验邮箱与账号一致**）。
- 同一邮箱不允许重复订阅。
- 用户默认不订阅，订阅后按钮切换为“取消订阅”。
- 当文章首次发布时，向所有有效订阅邮箱发送文章简介与文章链接。

## 2. 实现范围总览

本次实现包含以下模块：

- 数据层：新增订阅表 `email_subscriptions`。
- API 层：新增 `/api/subscriptions`（查询状态/订阅/取消订阅）。
- 前端层：改造 `BlogSidebar` 订阅卡片，支持登录态与按钮状态切换。
- 通知层：文章发布时触发订阅邮件发送。
- 验证层：访客发码/填码订阅与退订、登录态 JWT 订阅/退订、发布通知、草稿转发布通知等回归测试。

## 3. 数据库设计

### 3.1 新增表

文件：`lib/db/schema.ts`

新增表：`email_subscriptions`

关键字段：

- `email`：订阅邮箱，`unique`，防止重复订阅。
- `userId`：可选，已登录用户订阅时关联用户 ID。
- `isActive`：是否有效订阅。
- `subscribedAt`：首次订阅时间。
- `unsubscribedAt`：取消订阅时间。

### 3.2 `email_verifications` 类型扩展（访客订阅验证码）

文件：`lib/db/schema.ts`

在原有 `register` / `reset_password` / `change_email` 基础上增加：

- **`subscription`**：访客确认订阅。
- **`subscription_unsubscribe`**：访客确认退订。

变更后需重新生成并执行迁移（与订阅表迁移相同命令）。

### 3.3 迁移

执行方式（已验证）：

```bash
pnpm db:generate
pnpm db:migrate
```

> 若未迁移，`/api/subscriptions` 会因表不存在而报错；若未应用含新枚举值的迁移，写入验证码时会因 **ENUM 不含新值** 失败。

### 3.4 生产已执行 `db-migrate` 仍订阅失败（排障）

现象：在服务器执行了 `docker compose ... --profile migrate run ... db-migrate`，前端订阅仍 500 / `DB_SCHEMA_OUTDATED` / 表不存在类错误。

**1）确认迁移命令真正成功结束**

成功时 Drizzle 会输出类似 **`migrations applied successfully`**，且 shell **退出码为 0**。若日志停在 `Reading config file` 即无下文，可能是复制不全、终端被掐断或任务失败；请重跑并执行：

```bash
docker compose --env-file deploy/.env.docker --profile migrate run --rm -T db-migrate; echo "exit=$?"
```

**2）最常见：迁移打在「Compose 里的 MySQL」，应用在连「另一台库」**

`db-migrate` 在 `docker-compose.yml` 里被强制使用 **`DB_HOST=mysql`**（容器网络内的库）。若 **`blog-web` 未用同一套 Compose 启动**，而是宿主机上 `pnpm start` / PM2，且 `.env` / `.env.production` 里 **`DB_HOST` 指向云数据库、内网其它 IP、`127.0.0.1` 映射到别的实例**，则迁移改的是 A 库，接口读写的是 B 库，B 上永远没有 `email_subscriptions`。

请在**运行 Next 的同一环境**核对连接参数，并与迁移目标对比：

- **应用在 Docker 里**：`docker exec blog-next-app sh -c 'env | grep -E "^DB_"'`
- **应用在宿主机**：查看实际加载的 `DB_HOST`、`DB_PORT`、`DB_NAME`、`DB_USER`（勿与 `deploy/.env.docker` 想当然混用）

然后在**你认为应用正在使用的那台 MySQL** 上查表（把密码、库名换成你的真实值）：

```bash
docker exec -i blog-mysql mysql -uroot -p"你的root密码" -e "USE blog_system; SHOW TABLES LIKE 'email_subscriptions'; SELECT * FROM drizzle_migrations ORDER BY id;"
```

若此处无 `email_subscriptions`，说明要么迁移没应用到这台实例，要么库名不是 `blog_system`。

**3）代码过旧：`drizzle/` 里没有 0001 迁移**

`db-migrate` 镜像只打包当前仓库的 `drizzle/*.sql`。若 `/opt/blog-next` 未 `git pull` 到含 **`0001_*`（email_subscriptions）** 的版本，迁移跑完也不会建订阅表。解决：更新代码后重新 **`docker compose build db-migrate`**（或带 `--build` 再 `run`），再执行 migrate。

**4）统一修复思路**

在**与线上 Next 进程 `DB_*` 完全一致**的那台库上执行迁移（或把应用 `DB_*` 改为与已迁移库一致），并确保该次部署的镜像/目录内含有对应 `drizzle` SQL。

## 4. API 设计与行为

文件：`app/api/subscriptions/route.ts`

### 4.1 查询订阅状态

- `GET /api/subscriptions?email=<邮箱>`
- 返回：`{ email, isSubscribed }`

用途：

- 已登录状态下初始化按钮显示（订阅/取消订阅）。

### 4.2 订阅

- `POST /api/subscriptions`
- 入参：`{ email, verificationCode? }`（`userId` 字段已废弃，服务端以 JWT 为准。）

规则：

- **已登录**：请求头 **`Authorization: Bearer <accessToken>`**；`email` 必须与当前用户账号邮箱一致，否则 **403**；**不需要** `verificationCode`。
- **未登录（访客）**：**必须**提交 **`verificationCode`**（须与已通过 **`POST /api/auth/send-verification-code`** 下发、类型为 **`subscription`** 的未过期未使用验证码匹配）；缺失时返回 **400**，`code` 可为 **`SUBSCRIPTION_VERIFICATION_REQUIRED`**。
- 邮箱格式校验，不合法返回 400。
- 若邮箱已存在且 `isActive=true`：返回成功并提示“已订阅，无需重复订阅”。
- 若邮箱已存在但已取消：重新激活该记录。
- 若邮箱不存在：新建订阅记录。

### 4.3 取消订阅

- `DELETE /api/subscriptions`
- 入参：`{ email, verificationCode? }`

规则：

- **已登录**：同订阅，须有效 JWT 且 `email` 与账号邮箱一致；**不需要** `verificationCode`。
- **访客**：**必须**提交 **`verificationCode`**（类型 **`subscription_unsubscribe`**）；缺失时 **400**，`code` 可为 **`SUBSCRIPTION_VERIFICATION_REQUIRED`**。
- 有效订阅记录置为 `isActive=false` 并记录 `unsubscribedAt`。
- 若本就未订阅，返回成功（幂等）。

### 4.4 访客验证码发送（复用认证发码接口）

- `POST /api/auth/send-verification-code`
- `type` 取 **`subscription`**：若该邮箱**已处于有效订阅**，返回 **409**，不重复发订阅确认码。
- `type` 取 **`subscription_unsubscribe`**：若该邮箱**当前未订阅**，返回 **400**，避免对任意邮箱发送退订验证码。

## 5. 前端交互设计

文件：

- `components/blog/blog-sidebar.tsx`
- `components/layout/footer.tsx`
- `lib/hooks/useNewsletterGuestSubscription.ts`（访客发码/防抖查状态/提交）

### 5.1 未登录用户

- 显示邮箱输入框、**验证码输入框**、**「发送验证码」**（订阅意图发 `type=subscription`，已订阅意图退订时发 `type=subscription_unsubscribe`）、主按钮。
- 主按钮在「未订阅」时提交 **`POST /api/subscriptions`**（带 `verificationCode`）；在「已订阅」时提交 **`DELETE /api/subscriptions`**（带 `verificationCode`）。
- 输入邮箱后防抖调用 **`GET /api/subscriptions`**，用于切换主按钮文案（订阅 / 取消订阅）及决定发码类型。

### 5.2 已登录用户

- 不显示邮箱与验证码输入框。
- 默认显示“订阅”按钮（使用登录邮箱）。
- 组件初始化调用 `GET /api/subscriptions` 查询当前邮箱订阅状态：
  - 未订阅：按钮显示“订阅”
  - 已订阅：按钮显示“取消订阅”
- 点击订阅/取消订阅时请求头携带 **`Authorization: Bearer`**，调用 `POST` / `DELETE` **`/api/subscriptions`**（**不带** `verificationCode`）。

### 5.3 关键点

- 登录邮箱来源：`useAuth()` 的 `user.email`。
- 提交中禁用/加载状态：避免重复提交。
- 订阅结果使用统一消息提示反馈。
- 「发送验证码」客户端 **60s** 冷却（与服务端 **1 分钟** 频控互补）；服务端拒绝时展示接口返回文案。

## 6. 发布通知链路

### 6.1 邮件发送函数

文件：`lib/utils/email.ts`

新增：

- `sendPostPublishedEmail({ to, title, excerpt, postUrl })`

内容：

- 发送文章标题、简介和“阅读全文”链接。

### 6.2 订阅服务

文件：`lib/services/subscription.service.ts`

新增：

- `notifyOnPostPublished(post)`

逻辑：

- 查询 `isActive=true` 的订阅邮箱。
- 逐个发送邮件，统计成功/失败数量。
- 输出发送汇总日志。
- 文章链接由 `NEXT_PUBLIC_APP_URL` 与 `/${lang}/blog/${slug}` 组合；当前服务默认使用 `zh-CN` 作为邮件链接语言前缀。

- `app/api/posts/route.ts`
- `app/api/posts/[id]/route.ts`

触发时机：

- 创建文章且状态为 `published`。
- 更新文章从“非 published”变为 `published`（`PUT`/`PATCH`）。

## 7. 本次实现中的关键修复点

### 7.1 发布触发判断修复（重要）

问题：

- 文章服务返回结构存在嵌套（`posts` 子对象），直接读取 `newPost.status` 可能失效。

修复：

- 统一使用 `const postCore = (result as any)?.posts || result` 再读取 `status/title/slug/excerpt`。

效果：

- 避免“实际发布了，但未触发订阅通知”的问题。

### 7.2 可观测性增强

新增日志：

- 单封邮件发送成功日志（含收件邮箱、`messageId`）。
- 单封邮件失败日志（含收件邮箱）。
- 批量发送汇总日志（总数/成功数/失败数）。
- 订阅接口在检测到库表缺失时返回 `503`，并携带 `code=DB_SCHEMA_OUTDATED`，用于快速识别“未迁移/迁移错库”问题。

作用：

- 便于快速定位“未收到邮件”到底是未触发、发送失败还是邮箱侧分流问题。

已完成并通过：

## 9. 部署与配置说明

### 9.1 邮件配置

参考变量：

- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_SECURE`
- `SMTP_USER`
- `SMTP_PASS`

若未正确配置，邮件发送阶段会失败（例如 SMTP 鉴权报错）。

### 9.2 链接域名/IP

订阅邮件中的文章链接来源：

- `NEXT_PUBLIC_APP_URL`（未配置时回退 `http://localhost:3000`）

无域名场景建议直接配置服务器公网 IP，例如：

```env
NEXT_PUBLIC_APP_URL=http://<SERVER_IP>:13001
```
