# 接口变更记录

本文件专门记录 **HTTP API（`app/api`）** 的变更历史，便于对接与回归。**新记录一律写在最上方**（时间倒序）。每次变更应同步更新 **`docs/接口文档.md`** 中的对应条目。

## 2026-04-21 — 订阅接口收敛：访客验证码 + 登录邮箱一致性校验 + 发布通知触发

**类型**：feat（api / subscriptions）

### 变更说明

- `POST /api/subscriptions` 与 `DELETE /api/subscriptions` 增加双通道规则：
  - 登录态：要求 Bearer 有效，且 `email` 必须与当前账号邮箱一致；
  - 访客：必须提交验证码（`subscription` / `subscription_unsubscribe`）。
- `POST /api/auth/send-verification-code` 补充订阅场景约束：
  - `type=subscription`：邮箱已订阅时返回 409，避免重复发码；
  - `type=subscription_unsubscribe`：邮箱未订阅时返回 400，避免对任意邮箱发退订验证码。
- 订阅相关接口在库表缺失场景统一返回 `503 + DB_SCHEMA_OUTDATED`，登录 token 失效返回 `401 + AUTH_TOKEN_INVALID`，便于联调排障。
- 文章发布通知链路补充：
  - 新建即发布（`POST /api/posts`）触发；
  - 从非 published 变为 published（`PUT/PATCH /api/posts/{id}`）触发。

### 涉及文件

- `app/api/subscriptions/route.ts`
- `app/api/auth/send-verification-code/route.ts`
- `lib/hooks/useNewsletterGuestSubscription.ts`
- `components/blog/blog-sidebar.tsx`
- `lib/services/subscription.service.ts`
- `app/api/posts/route.ts`
- `app/api/posts/[id]/route.ts`

### 对客户端影响

- 访客订阅/退订必须先发码并携带验证码提交；登录用户不再需要验证码，但邮箱必须与当前账号一致。
- 客户端可按 `DB_SCHEMA_OUTDATED` 与 `AUTH_TOKEN_INVALID` 做更明确的错误分流提示与恢复引导。

## 2026-04-21 — 个人资料接口收敛：邮箱改绑验证码校验 + 社交字段规范化

**类型**：feat（api / profile）

### 变更说明

- `POST /api/profile` 与 `PUT /api/profile` 在修改邮箱场景下，统一要求先完成 `type=change_email` 验证码校验（`emailVerificationCode`）。
- `GET /api/profile` 返回数据以 `users` 表为主回填 `email`、`avatar`、`displayName`，解决资料页与全站用户信息不一致问题。
- `socialLinks` 统一收敛为 `github`、`wechatQr`、`douyin`、`bilibili`，并兼容历史键名（如 `wechat_qr`、`wechat`）读取，降低历史数据迁移成本。
- 资料更新成功后会写入 `user_activities`（`profile_updated`），支持个人中心“最近活动”回溯。

### 涉及文件

- `app/api/profile/route.ts`
- `components/profile/profile-settings.tsx`
- `components/profile/profile-overview.tsx`
- `components/profile/profile-activities.tsx`
- `types/blog.ts`

### 对客户端影响

- 客户端在修改邮箱时，必须先调用 `POST /api/auth/send-verification-code`（`type=change_email`）再提交资料保存。
- 社交资料渲染与编辑应使用当前四个键（`github`、`wechatQr`、`douyin`、`bilibili`），避免继续写入已弃用字段（如 `twitter`、`linkedin`、`weibo`）。

## 2026-04-19 — 密码保护文章支持首页/博客列表展示与前置解锁

## 2026-04-17 — 分类/标签接口改为登录必需 + owner 级数据隔离

**类型**：refactor（api / auth / data-scope）

### 变更说明

- `GET/POST /api/categories` 与 `GET/PUT/DELETE /api/categories/{id}` 统一要求 `Authorization: Bearer`。
- `GET/POST /api/tags` 与 `GET/PUT/DELETE /api/tags/{id}` 统一要求 `Authorization: Bearer`。
- 分类与标签接口查询/更新/删除均加入 `ownerId = 当前登录用户` 条件，实现用户级数据隔离。
- 创建分类/标签时，`ownerId` 由服务端根据登录态写入，不接受客户端指定。

### 涉及文件

- `app/api/categories/route.ts`
- `app/api/categories/[id]/route.ts`
- `app/api/tags/route.ts`
- `app/api/tags/[id]/route.ts`
- `lib/utils/api-docs-endpoint-meta.ts`
- `docs/接口文档.md`

### 对调用方影响

- 未登录访问分类/标签接口将返回 401。
- 客户端若需要读取分类/标签，必须携带 Bearer；并且只能读取到当前账号自己的数据。

## 2026-04-17 — PATCH 管理用户：禁止修改根账户角色/状态

**类型**：feat（安全约束 + 前台）

### 变更说明

- **`PATCH /api/admin/users/{id}`**：当路径 `id` 与当前超级管理员令牌中的 `userId`（根账户）相同时，返回 **403**，不修改数据库。
- **账户管理弹窗**：目标用户为根账户（详情中 `role` 为展示用的 `super_admin`）时，不再显示「角色」「状态」下拉，仅只读展示；「保存」禁用。
- 修正打开详情时曾将 `super_admin` 误映射为 `admin` 的状态，避免误发起 PATCH。

### 涉及文件

- `app/api/admin/users/[id]/route.ts`
- `components/profile/profile-accounts-admin.tsx`
- `lib/utils/api-docs-endpoint-meta.ts`、`docs/接口文档.md`

### 对调用方影响

- 自动化脚本若对根账户本人发 PATCH，将收到 403；正常仅改其他用户时不影响。

## 2026-04-17 — 超级管理员与互动接口去兼容化清理

**类型**：refactor（auth / api）

### 变更说明

- 超级管理员统一为真实 DB 用户模型，不再依赖 `userId=0` 约定。
- `getAuthUserFromRequest` 改为仅接受正整数 `userId`，删除旧 token 特判。
- 点赞/收藏/engagement 删除 `userId=0` 限制分支，登录用户（含 DB 超管）均按同一逻辑处理。
- 注册、资料邮箱冲突校验、关于页站长资料统一按真实超级管理员 `user_id` 查询。

### 涉及文件

- `lib/config/super-admin.ts`、`lib/utils/request-auth.ts`
- `app/api/posts/engagement/route.ts`、`app/api/posts/[id]/like/route.ts`、`app/api/posts/[id]/favorite/route.ts`
- `app/api/auth/login/route.ts`、`app/api/auth/register/route.ts`、`lib/services/profile-email.service.ts`、`lib/services/about-owner.service.ts`

### 对调用方影响

- 不再支持历史 `userId=0` 会话；需重新登录获取新版 token。
- 超级管理员与普通登录用户在互动接口上行为一致。

## 2026-04-17 — 超级管理员改为真实 DB 用户

**类型**：fix（auth / api）

### 变更说明

- `resolveSuperAdminLogin` 改为登录成功后自动确保 `users` 超管行存在（`role=admin`），并签发 `role=super_admin + isRoot=true` 但 `userId` 使用真实 DB id。
- `refresh` 支持旧 token平滑刷新为新 token（真实 DB `userId`）。
- 超管 profile 自动迁移：若仅存在 `user_profiles.user_id=<super_admin_id>`，会在首次登录时复制到真实超管 `user_id`。
- 点赞/收藏/engagement 的 限制仅对 `userId=0` 生效；新版 DB 超管可正常使用互动功能。

### 涉及文件

- `lib/config/super-admin.ts`、`app/api/auth/login/route.ts`、`app/api/auth/refresh/route.ts`
- `app/api/posts/engagement/route.ts`、`app/api/posts/[id]/like/route.ts`、`app/api/posts/[id]/favorite/route.ts`
- `app/api/auth/register/route.ts`、`lib/services/profile-email.service.ts`、`lib/services/about-owner.service.ts`

### 对调用方影响

- 超管登录后获得真实 `userId`，可参与点赞/收藏；旧 refresh token 可无感升级。
- 历史 `userId=0` 相关兼容逻辑已移除，互动接口统一按真实数据库用户身份处理。

## 2026-04-17 — 文章互动接口：JWT 归一化、超级管理员约束与点赞落库健壮性

**类型**：fix（行为细化 + 排错增强）

### 变更说明

- **`getAuthUserFromRequest`**：`userId` 支持数字字符串归一化；拒绝非法的 `userId=0`（仅允许与刷新令牌一致的超级管理员：`role=super_admin` 且 `isRoot=true`）。
- **`GET /api/posts/engagement`**：查询结果 `postId` 统一 `Number` 以兼容驱动 `bigint`；失败时 `console.error` 附带 MySQL 驱动错误解析。
- **`POST /api/posts/{id}/like`**：超级管理员 **403**；`user_post_likes` 插入显式 `created_at`；`like_count` 改为读改写并经 `toJsonSafeInt` 输出，避免 `BigInt` 序列化异常及表达式默认值差异。
- **`POST /api/posts/{id}/favorite`**：超级管理员 **403**；插入收藏显式 `created_at`；失败日志增强。

### 涉及文件

- `lib/utils/request-auth.ts`、`lib/utils/tool.ts`（`toJsonSafeInt`）
- `app/api/posts/engagement/route.ts`、`app/api/posts/[id]/like/route.ts`、`app/api/posts/[id]/favorite/route.ts`
- `lib/utils/api-docs-endpoint-meta.ts`、`docs/接口文档.md`

### 对调用方影响

- 使用 **超级管理员**（数据库用户）时：互动接口按真实用户身份执行。
- 普通用户契约不变；若此前因 `created_at` / 计数序列化导致 500，应随本次修复消除。
