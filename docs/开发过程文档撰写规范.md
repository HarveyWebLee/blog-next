# 开发过程文档撰写规范

本文说明在本仓库中，**如何**将与 Cursor AI 协作产出的需求、方案与实现沉淀到 `docs/`，并与 [文档索引.md](./文档索引.md) 配合使用。

## 何时必须写/改文档

- 实现**新功能**、**非平凡配置**（Husky、ESLint、环境变量约定等）或**架构级调整**。
- **用户需求发生变化**：在同一轮代码修改中，更新对应文档章节，避免「代码已变、文档仍写旧行为」。
- **修复影响面较大的 Bug**：建议简短记录现象、根因、修复要点与涉及路径，便于回归。
- **涉及接口或数据表变更**：当改动 `app/api` 或 `lib/db/schema.ts` 时，除专题文档外，还需同步 `docs/接口文档.md`、`docs/数据表变更文档.md` 等，并遵守 `.cursor/rules/接口与数据库文档同步.mdc`。

纯只读答疑且未改仓库时，可按协作规则选择不建档；用户明确要求不写文档时除外。

## 文档应包含什么

建议每份任务或专题文档至少覆盖：

1. **背景与目标**：用户原话需求提炼、约束（浏览器、环境、不做什么）。
2. **方案摘要**：选了哪条路、为什么；未选方案可一句话带过。
3. **关键实现线索**（优先可检索）：
   - 文件路径（如 `app/[lang]/foo/page.tsx`、`lib/utils/bar.ts`）
   - **重要导出**、路由、`package.json` 脚本、环境变量名
   - 配置片段：只抄**必要几行**，其余用路径引用
4. **验证方式**：如何手动或命令行确认（如 `pnpm lint`、`pnpm dev` 下某路径）。
5. **后续事项**（可选）：债项、待办、与 `功能清单与路线图.md` 的衔接。

避免整页粘贴与当前仓库一致的冗长源码；用路径 + 摘要即可。

## 文件命名与放置

- 位置：**一律 `docs/`**（根目录保留 `README.md` 作为项目门面即可）。
- **文件名用中文**（与仓库现有风格一致），例如：`某某功能说明.md`。
- **入库级变更流水**：一律追加到 **`docs/变更记录.md` 的最上方**（格式见该文件文首说明），**不要**再为每次变更单独新建 `变更记录-日期-标题.md`。
- 若长期从事同一专题，优先**更新已有文档**而非无限新增；过时章节删除或标注 **「已废弃」** 并写替代文档链接。

## 索引维护

- 新建或废弃文档后，更新 **[文档索引.md](./文档索引.md)** 中对应表格或章节，避免失联。
- 若某文档被多篇引用，改动行为后检查文中交叉引用是否仍有效。

## Cursor 规则入口

- 强制约定见项目内 **`.cursor/rules/AI协作与文档沉淀.mdc`**。

**版本**：与仓库当前协作规范同步；规则变更时请同时更新本页与 `.cursor/rules` 内相关文件。
