# 认证实现说明（按当前代码）

## 概述

当前认证体系基于 JWT + `bcryptjs`，覆盖登录、注册、验证码、刷新令牌等能力，并支持“内存态超级管理员 root 会话”。

## 已实现能力

### 登录、刷新与登出

- `POST /api/auth/login`：支持用户名或邮箱登录。
- 登录成功后：
  - 响应体返回 `user` 与 `token`（access）；
  - 同时写入 HttpOnly Cookie `blog_refresh_token`（`SameSite=Lax`、`Path=/api/auth`，有效期对齐 `JWT_REFRESH_EXPIRES_IN`）。
  - **Secure 标志**：默认跟随对外协议（`X-Forwarded-Proto` / 请求 URL）；可用 `AUTH_COOKIE_SECURE=true|false` 覆盖。无 SSL 的 HTTP 生产访问不加 `Secure`，避免浏览器丢弃 Cookie。
- `POST /api/auth/refresh`：**仅**读取 Cookie；须校验 Origin/Referer（同站或 `CORS_ORIGIN`）。成功响应仅含新的 `token`，并轮换 Cookie。
- `POST /api/auth/logout`：清除 refresh Cookie。
- 前台 `AuthProvider`：`credentials: "include"`；**不**将 refresh 写入 `localStorage`（启动时清理历史残留）；accessToken 经 `client-bearer-auth` 写入。
- 普通数据库用户登录会校验 `status === active`，非激活用户返回 403。
- **密码传输**：HTTPS 下可强制 `passwordTransport`（Web Crypto）；**明文 HTTP** 请求自动放宽为允许明文（浏览器非安全上下文无 `crypto.subtle`）。

### 注册与验证码

- `POST /api/auth/register`：支持常规注册；`useEmailVerification=true` 时需校验 `register` 验证码。
- `POST /api/auth/send-verification-code`：支持五种验证码类型（`register`、`reset_password`、`change_email`、`subscription`、`subscription_unsubscribe`）。
- `POST /api/auth/verify-code`：校验并消费验证码。

### 忘记/重置密码

- `POST /api/auth/forgot-password`：
  - 支持 body 或 query 传入 `email`；
  - 校验邮箱格式与注册状态；
  - 使用 Redis 优先、数据库回退的 10 分钟 3 次限流；
  - 生成一次性重置 token，写入 `email_verifications`（`type=reset_password`，30 分钟过期），并失效同邮箱旧 token；
  - 按用户资料语言发送密码重置邮件；
  - 记录 `PASSWORD_RESET_REQUESTED` 用户活动。
- `POST /api/auth/reset-password`：
  - 支持 `passwordTransport` 传输封装，也兼容灰度期明文字段；
  - 校验 token 存在、未使用、未过期；
  - 校验新密码强度，更新 `users.password`，消费 token；
  - 记录 `PASSWORD_RESET_COMPLETED` 用户活动。

### 超级管理员 root 会话

- 配置来源：`lib/config/super-admin.ts` + `env.example` 中 `SUPER_ADMIN_*`。
- 登录成功后会确保存在真实 `users` 超管行（数据库 **`users.role` 存 `admin`**），并签发 JWT 中 **`role=super_admin`、`isRoot=true`**（见 `lib/config/super-admin.ts`）。鉴权以 JWT 语义角色为准，非 DB 枚举列。
- `refresh` 对该会话同样生效，并统一回到真实 DB `userId`。

## 页面路由与布局

- 认证页面路径：`/{lang}/auth/login`、`/{lang}/auth/register`、`/{lang}/auth/forgot-password`、`/{lang}/auth/reset-password`。
- 认证页通过 `components/layout/conditional-layout.tsx` 隐藏全站 Header/Footer，使用独立全屏背景布局。

## 环境变量（认证相关）

- `DB_*`
- `JWT_SECRET`、`JWT_REFRESH_SECRET`
- `JWT_EXPIRES_IN`、`JWT_REFRESH_EXPIRES_IN`（refresh Cookie `maxAge` 与此对齐）
- `AUTH_COOKIE_SECURE`（可选：`true`/`false`/`auto`；默认 auto）
- `CORS_ORIGIN`（独立前端 Origin 白名单；Cookie 跨域需配合 `Access-Control-Allow-Credentials`）
- `PASSWORD_TRANSPORT_*`（HTTPS 下可强制封装；明文 HTTP 自动放宽）
- `SUPER_ADMIN_*`（可选）
- `SMTP_*`（验证码邮件）

以 `env.example` 为准，不使用历史 `DATABASE_URL` / `NEXTAUTH_*` 口径。

## 本地运行

```bash
pnpm install

```

## 风险与后续建议

- 生产环境需确保 `NEXT_PUBLIC_APP_URL`、`SMTP_*`、`PASSWORD_TRANSPORT_*` 配置正确，避免重置链接域名错误或密码传输加固未生效。
- refresh 已统一为 Cookie；依赖 Body / JSON `refreshToken` 的外部脚本需改为 `credentials: "include"`。
- accessToken 仍短期存于 `localStorage`（经统一读写入口），业务组件应使用 `clientApiFetch` / `clientBearerHeaders`，勿直接读存储。
- 对登录与验证码接口增加更细粒度限流与审计事件（如失败次数、IP 维度统计）。
