## 2026-04-10 — 邮件订阅与文章发布通知功能

**类型**：功能 / API / 邮件通知

### 需求

- 未登录用户可输入邮箱订阅更新。
- 已登录用户使用注册邮箱订阅/取消订阅，不显示邮箱输入框。
- 同一邮箱不可重复订阅。
- 文章发布时向所有有效订阅邮箱发送文章简介与文章链接。

### 处理

- 数据层：新增 `email_subscriptions`（`email` 唯一、`isActive`、`subscribedAt`、`unsubscribedAt`、`userId` 可选）。
- API：新增 `app/api/subscriptions/route.ts`，提供状态查询（GET）、订阅（POST）、取消订阅（DELETE）。
- 前端：改造 `components/blog/blog-sidebar.tsx` 的订阅卡片：
  - 未登录显示邮箱输入框 + 订阅按钮；
  - 已登录仅显示按钮，并根据订阅状态切换“订阅/取消订阅”。
- 通知链路：新增 `lib/services/subscription.service.ts` 与 `sendPostPublishedEmail`，在：
  - `app/api/posts/route.ts`（新建即发布）
  - `app/api/posts/[id]/route.ts`（PUT/PATCH 从未发布到已发布）
    触发订阅通知发送。
- 修复发布触发判断：兼容文章服务返回嵌套结构（`posts` 子对象）。
- 增强日志：输出邮件发送成功/失败与批量发送汇总。

### 验证

- 已实测匿名订阅、登录订阅、重复订阅防重、取消订阅、发布触发通知、草稿转发布触发通知、取消订阅后不再通知。

## 2026-04-10 — 修复 semantic-release 提交时 Husky 拦截 CHANGELOG

**类型**：CI / semantic-release

### 问题

`@semantic-release/git` 在 CI 中执行 `git commit` 时仍会安装并触发 **husky** `pre-commit`（`format:check`）。**`@semantic-release/changelog`** 生成的 **`CHANGELOG.md`** 与 Prettier 规则不一致，导致提交失败。

### 处理

1. **`@semantic-release/exec`**：在 **changelog、npm** 插件之后，**git** 提交之前执行 **`prettier --write CHANGELOG.md package.json`**。
2. **`.github/workflows/release.yml`**：在 **semantic-release** 步骤设置 **`HUSKY=0`**，避免发布机器人提交再走本地钩子（与 Prettier 双保险）。

## 2026-04-10 — GitHub Actions 自动发版（semantic-release）

**类型**：CI / 工程自动化

### 需求

推送到 GitHub **`main`** 且存在可发布提交（如 **`feat`**、**`fix`**、**`perf`**）时，由 CI 运行 **semantic-release**：提升根包 **`version`**、更新 **`CHANGELOG.md`**、创建 **GitHub Release**。提交信息符合仓库 **Angular** 约定（`type(scope): 中文说明`）。本地 **`pnpm run release:dry-run`** 预演。

### 方案

- 安装 **`semantic-release`** 及官方插件（changelog、npm 仅 bump、github、git、analyzer、release-notes-generator）。
- 新增 **`release.config.cjs`**：`private` 包 `npmPublish: false`；`@semantic-release/git` 提交 **`[skip ci]`** 以免循环触发。
- 新增 **`.github/workflows/release.yml`**：`push` → `main`，`pnpm install --frozen-lockfile` 后 `semantic-release`；`GITHUB_TOKEN` + `contents: write`。
- **`package.json`**：增加 **`repository`**、`script` **`release:dry-run`**。

### 关键路径

`release.config.cjs`、`.github/workflows/release.yml`、`package.json`、`docs/自动发版与semantic-release.md`

### 验证

- `pnpm run release:dry-run`：需 **git 远程与完整历史**；若要在本地跑通 GitHub 插件校验，需临时设置 **`GITHUB_TOKEN`** / **`GH_TOKEN`**（见 `docs/自动发版与semantic-release.md`）。
- 合并到 `main` 后，在 **Actions** 中可见 **Release**；有可发布提交时应出现新 tag 与 Release。

### 说明

仓库为**单根包**发版；`.cursorrules` 中「GitLab CI」已改为 **GitHub Actions** 表述以与实现一致。

## 2026-04-10 — Git 主分支使用 main

**类型**：Git / 工程约定

### 需求

将仓库默认主分支从 `master` 改为 **`main`**。

### 已做操作（本地）

1. **`git branch -m main`**：将当前分支名由 `master` 重命名为 `main`（本仓库当时尚无任何 commit 时亦可执行）。
2. **`git config init.defaultBranch main`**：本仓库本地配置中指定 **`git init` 时默认初始分支为 `main`**。

### 文档与规则

- **`.cursorrules`** 中 semantic-release 相关表述为仅写推送到 **`main`**。

### 远程仓库（若已存在）

若 GitLab/GitHub 上仍只有 `master`，首次 `git push` 后需在托管平台将**默认分支**设为 `main`，并按需删除或归档 `master`。

```bash
git push -u origin main
```

### 验证

- `git status -sb` 显示在 **`main`** 上。
