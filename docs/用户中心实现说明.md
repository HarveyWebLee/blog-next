# 个人中心功能实现文档

本文档描述了个人中心功能的完整实现，包括数据库设计、API接口、UI组件和国际化支持。

> 说明：本文中的 SQL 结构与示例请求用于帮助理解，权威实现以 `lib/db/schema.ts` 与 `app/api/profile/**` 实际代码为准。

## 数据库设计

### 相关表结构（当前实现）

> **权威来源**：`lib/db/schema.ts` 与 Drizzle 迁移。下文 SQL 为概念说明，字段类型、可空性与 JSON 列以 schema 为准（例如 `user_activities.user_id` **可空**以支持访客活动；`user_profiles` 含 `email` 等扩展字段）。

#### 1. 用户个人资料表 (user_profiles)

见 `lib/db/schema.ts` 中 `userProfiles` 定义，主要含 `userId`、姓名、`email`（改绑同步）、`displayName`、`avatar`、`socialLinks`、`privacy`、`notifications` 偏好等。

#### 2. 用户活动日志表 (user_activities)

见 `userActivities`：`userId` 可空、`action`、`description`、`metadata`（JSON）、`ipAddress`、`userAgent`、`createdAt`。

#### 3. 用户收藏表 (user_favorites)

复合业务：`userId` + `postId`，见 `userFavorites`。

#### 4. 用户点赞表 (user_post_likes)

复合主键 `(user_id, post_id)`，见 `userPostLikes`。

#### 5. 用户关注表 (user_follows)

`followerId` → `followingId`，见 `userFollows`。

#### 6. 用户通知表 (user_notifications)

见 `userNotifications`：`type`（comment/like/follow/mention/system 等）、`title`、`content`、`data`、`isRead`、`readAt`。

## API接口设计

### 个人资料API

#### GET /api/profile

获取当前用户个人资料

**响应示例:**

```json
{
  "data": {
    "id": 1,
    "userId": 1,
    "firstName": "张",
    "lastName": "三",
    "phone": "+86 138 0013 8000",
    "website": "https://example.com",
    "location": "北京市",
    "displayName": "张三",
    "email": "demo@example.com",
    "avatar": "https://cdn.example.com/profile/avatar.jpg",
    "notifications": {
      "email": true,
      "push": true,
      "sms": false
    },
    "privacy": {
      "profileVisibility": "public",
      "emailVisibility": "private"
    },
    "socialLinks": {
      "github": "https://github.com/username",
      "wechatQr": "https://cdn.example.com/profile/wechat-qr.png",
      "douyin": "douyin_id",
      "bilibili": "https://space.bilibili.com/123456"
    }
  },
  "message": "个人资料获取成功",
  "success": true,
  "timestamp": "2024-01-01T00:00:00.000Z"
}
```

#### POST /api/profile

首次创建当前登录用户的 `user_profiles` 资料行。

#### PUT /api/profile

更新个人资料

**请求体:**

```json
{
  "avatar": "https://cdn.example.com/profile/avatar.jpg",
  "displayName": "张三",
  "email": "new@example.com",
  "emailVerificationCode": "123456",
  "firstName": "张",
  "lastName": "三",
  "location": "北京市",
  "notifications": {
    "email": true,
    "push": true,
    "sms": false,
    "comment": true,
    "like": true,
    "follow": true
  },
  "phone": "+86 138 0013 8000",
  "privacy": {
    "profileVisibility": "public",
    "emailVisibility": "private",
    "activityVisibility": "public"
  },
  "socialLinks": {
    "github": "https://github.com/username",
    "wechatQr": "https://cdn.example.com/profile/wechat-qr.png",
    "douyin": "douyin_id",
    "bilibili": "space_url_or_uid"
  },
  "website": "https://example.com"
}
```

> 说明：
>
> - 当 `email` 与当前邮箱不一致时，必须先通过 `POST /api/auth/send-verification-code`（`type=change_email`）获取验证码，再提交 `emailVerificationCode`。
> - 后端会将 `avatar` / `displayName` / `email` 同步回 `users` 表，保证全站头部用户信息一致。

### 收藏管理API

#### GET /api/profile/favorites

获取用户收藏列表

**查询参数:**

- `page`: 页码 (默认: 1)
- `limit`: 每页数量 (默认: 10)

#### POST /api/profile/favorites

收藏文章

**请求体:**

```json
{
  "postId": 123
}
```

#### DELETE /api/profile/favorites?postId=123

取消收藏

### 点赞管理 API

#### GET /api/profile/likes

获取当前用户点赞的文章列表。

**查询参数:**

- `page`: 页码 (默认: 1)
- `limit`: 每页数量 (默认: 10)

#### DELETE /api/profile/likes?postId=123

取消对指定文章的点赞。

### 通知管理 API

> **前台通知中心**（`components/profile/profile-notifications.tsx`）使用 **`/api/notifications`** 作为主路径。`/api/profile/notifications` 为并行遗留接口，行为类似，详见 [个人中心社交与通知模块功能文档.md](./个人中心社交与通知模块功能文档.md)。

#### GET /api/notifications

获取用户通知列表

**查询参数:**

- `page`: 页码
- `limit`: 每页数量（通知中心 UI 使用 20）
- `type`: 通知类型 (comment, like, follow, mention, system)
- `isRead`: 是否已读 (true/false)

#### PUT /api/notifications

标记通知为已读

**请求体:**

```json
{
  "markAllAsRead": false,
  "notificationIds": [1, 2, 3]
}
```

#### DELETE /api/notifications

删除通知。Body 示例：`{ "clearRead": true }` 清理全部已读，或 `{ "notificationIds": [1, 2] }` 批量删除。

#### GET /api/profile/notifications（遗留）

与上组接口并行，参数与语义相近；新功能优先 `/api/notifications`。

### 活动日志API

#### GET /api/profile/activities

获取用户活动日志

**查询参数:**

- `page`: 页码
- `limit`: 每页数量
- `action`: 活动类型

#### POST /api/profile/activities

记录用户活动

**请求体:**

```json
{
  "action": "post_created",
  "description": "创建了文章《如何学习React》",
  "metadata": {
    "postId": 123,
    "postTitle": "如何学习React"
  }
}
```

### 设计特点

1. **与博客前台一致**: 使用与 `BlogPageContent` 相同的页面渐变背景、玻璃拟态卡片与 `foreground` / `default-*` 语义色，避免独立「后台风」侧栏与顶栏。
2. **响应式设计**: 支持桌面端和移动端（小屏侧栏为抽屉，不重复站点 Header 的搜索与通知）。
3. **主题支持**: 明暗主题切换
4. **加载状态**: 与全站骨架色一致的加载占位
5. **空状态**: 友好的空状态提示
6. **交互反馈**: 按钮与链接与博客筛选区风格对齐
7. **权限分层导航**: 超级管理员在侧栏额外看到「账户管理」与「API 文档」入口

## 权限控制

### 认证要求

- 所有个人中心页面都需要用户登录
- 使用 JWT token 进行身份验证
- 未登录用户在 **`ProfileLayout`**（`components/profile/profile-layout.tsx`）中通过 `useAuth()` 检测后 **`router.replace(\`/${lang}/auth/login\`)`**

> 说明：仓库中存在 `middleware/profile.ts` 示例中间件，但**未被**根 `middleware.ts` 引用；服务端路由级门禁以各 API 的 `requireAuthUser` 为准。

### API 权限

- 各 `/api/profile/*` 与 `/api/notifications` 接口需 **Bearer** 令牌
- 用户只能访问自己的数据（超级管理员在部分全局通知接口可按 `userId` 跨用户操作）

## 使用指南

### 1. 数据库迁移

```bash
# 生成迁移文件
pnpm db:generate

# 执行迁移
pnpm db:migrate
```

### 2. 启动开发服务器

```bash
pnpm dev
```

### 3. 访问个人中心

```
http://localhost:3000/zh-CN/profile
```

### 4. API测试

```bash
# 获取个人资料
curl -H "Authorization: Bearer YOUR_TOKEN" \
     http://localhost:3000/api/profile

# 获取统计信息
curl -H "Authorization: Bearer YOUR_TOKEN" \
     http://localhost:3000/api/profile/stats

# 修改邮箱前发送验证码（change_email）
curl -X POST "http://localhost:3000/api/auth/send-verification-code" \
  -H "Content-Type: application/json" \
  -d '{"email":"new@example.com","type":"change_email"}'
```

## 部署注意事项

### 1. 环境变量

确保以下环境变量已正确配置（以根目录 `env.example` 为准）：

- `JWT_SECRET`: JWT 密钥
- `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME`: 数据库连接配置
- `SMTP_HOST` / `SMTP_PORT` / `SMTP_USER` / `SMTP_PASS`: 邮箱验证码发送配置（修改邮箱时必需）

### 2. 数据库权限

确保数据库用户具有以下权限：

- SELECT, INSERT, UPDATE, DELETE
- CREATE, ALTER (用于迁移)

### 3. 文件上传

如果支持头像上传，需要配置：

- 文件存储服务 (如AWS S3, Cloudinary)
- 文件大小限制
- 文件类型验证

### 1. 输入验证

- 所有用户输入都需要验证和清理

### 2. 权限控制

- 用户只能访问自己的数据

- 敏感信息加密存储

- 遵循GDPR等隐私法规

## 故障排除

### 常见问题

1. **认证失败**
   - 检查JWT token是否有效
   - 确认token未过期
   - 验证请求头格式

2. **数据库连接错误**
   - 检查 `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME`（见 `lib/db/config.ts`）
   - 确认数据库服务运行状态
   - 验证用户权限

3. **页面加载缓慢**
   - 检查数据库查询性能
   - 优化图片资源
   - 使用CDN加速
