# Feature Doc Closure

## 适用场景

- 新功能或非平凡配置落地后。
- 架构调整、复杂排障、影响面较大的 Bug 修复后。

## 不处理范围

- 不负责接口明细与接口变更流水（交给 `api-contract-sync`）。
- 不负责数据表清单与迁移流水（交给 `db-schema-sync`）。
- 不负责代码质量命令执行（交给 `precommit-quality-gate`）。

## 执行步骤

1. 识别文档落位
   - 优先更新已有专题文档；无对应文档再新增中文文件名文档。
2. 写入关键信息
   - 需求背景、方案取舍、关键文件路径、配置键、验证方式。
3. 更新索引
   - 新增或废弃专题文档时，同步 `docs/文档索引.md`。
4. 追加统一变更记录
   - 在 `docs/变更记录.md` 顶部追加日期与标题块。
5. 与代码同轮提交
   - 避免代码已改而文档滞后。

## 完成标准

- 文档与当前代码行为一致，且可从索引快速定位。
- 不生成与当前需求无关的总结文档。

## 最小使用示例

- 示例触发语句：`这个功能做完后，把实现方案和验证步骤沉淀到 docs`
- 预期行为：优先更新既有专题文档并维护索引与统一变更记录。
